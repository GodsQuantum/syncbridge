Synchronisation
Copie un dossier A → B (rsync/rclone). Le cas d'origine.
Commande / script
Lance une commande ou un script (ex. docker system prune -f, /import/scripts/CRON/DIAG-LIGHT.sh). Déclenchable en manuel, cron ou surveillance.
Exécutée via sh -c dans le conteneur. stdout/stderr capturés et streamés dans les logs (bouton « Lancer »). Job Docker : la socket doit être montée (voir compose).
Backend SyncBridge
SyncBridge exécute le job : logs en direct, bouton d'arrêt, verrou anti-chevauchement. S'arrête si le conteneur s'arrête.
Backend Système
Écrit un vrai cron/systemd sur l'hôte : le job survit à une panne Docker. Demande un déclencheur cron ou surveillance (pas manuel) + des montages hôte en écriture. Pas de logs live.
Tue le job s'il dépasse ce temps (anti-blocage). 0 = pas de limite.

Accumulation
Ajoute les nouveaux fichiers et met à jour les modifiés. Ne supprime jamais dans B. Le plus sûr.
Miroir exact
B devient une copie identique de A. Supprime dans B ce qui a disparu de A. Attention aux suppressions.
Déplacement
Copie vers B puis vide la source A. Pour évacuer un dossier de transit.
Manuel
Se lance uniquement quand tu cliques « Lancer maintenant ».
Planifié
Se lance à heure fixe via une expression cron.
Constant
Surveille A en continu et sync dès qu'un fichier change (attend 3 s de calme).

Format : min heure jour mois jour-semaine. Le * = "chaque".
Le dossier surveillé : un changement dedans déclenche la commande.
Ne déclenche que si un fichier correspond. Vide = tout. Sépare par des virgules.
Sur un montage NFS, inotify ne voit pas les écritures distantes : garde Hybride ou Scan.
Attend N s de calme après le dernier changement avant de lancer. Coalesce les rafales.
Fréquence du scan (modes Hybride/Scan). 300 = 5 min, comme un watcher classique.
Pour du local-to-local, rsync suffit et va vite. rclone utile si un jour tu vises un cloud.
Date : compare taille et horodatage. Checksum : lit le contenu, détecte la corruption, mais relit tout.
▸ Options avancées (protection & débit)
Backup système fidèle 🔒 Copie exacte : préserve propriétaires, permissions, ACL et hardlinks (flags -aHAX --numeric-ids --fake-super). Les fichiers restent en 1000 sur la destination, mais les vraies identités root sont stockées dans les xattr — ce qui rend le backup restaurable à l'identique (resurrection d'un serveur). Nécessite le conteneur en root (voir compose). Sans cette option, tout est copié en 1000:1000 simple.
Corbeille de sécurité Avant d'écraser ou supprimer dans B, déplace l'ancienne version dans un dossier daté .sb-backup/<date>. Filet de sécurité contre les erreurs.
Nombre de dossiers .sb-backup/ gardés. À chaque run, les plus anciens au-delà sont purgés. 0 = tout garder. Ex. 5 = un versioning des 5 dernières exécutions.
Ne pas écraser les fichiers plus récents dans B Si un fichier dans B est plus récent que dans A, on le garde. Utile si plusieurs sources alimentent une même destination.
Bride la vitesse pour ne pas saturer le disque/réseau. 10M = 10 Mo/s. Vide = illimité.
En mode miroir, abandonne si plus de N fichiers seraient supprimés. Protège d'un effacement massif accidentel. 0 = désactivé.
Motifs à ignorer, séparés par des virgules. Ex. fichiers système NAS (@eaDir), temporaires (*.tmp).
